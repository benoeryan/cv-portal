# Google Drive OCR fallback for scanned PDF date extraction

The certificate date extraction endpoint gains a second-pass OCR strategy for image-based PDFs. When `pdf-parse` returns empty text (indicating a scanned document), the route now uses the Google Drive API to copy the file as a Google Doc (triggering server-side OCR), exports the doc as plain text, extracts dates from that text, and cleans up the temporary doc. The approach mirrors the user's existing Apps Script workflow. A new `googleapis` dependency provides the Drive client, authenticated via a service account credential stored in `GOOGLE_SERVICE_ACCOUNT_JSON`.

**Watch for:** The OCR path requires the service account to have access to the original PDF file on Drive — if sharing isn't configured, the error message won't explain the permissions gap (confirmed). Credential JSON is parsed per-request with no caching, which will create auth overhead at scale (likely). The `drive` scope grants full access to all files the service account can see, broader than strictly necessary if the account accumulates shared files (confirmed).

## High-level view

The implementation splits the route into two extraction paths: fast text-based extraction via `pdf-parse`, and a Drive-OCR fallback when text extraction returns nothing. The date-finding logic is refactored into a shared `findDatesInText` helper used by both paths.

The OCR function authenticates a service account, copies the target PDF as a Google Doc to trigger OCR, exports the text, then deletes the temporary doc in a `finally` block. However, the `drive` scope grants full access to all files the service account can see, which is broader than the minimum needed if the account accumulates shared files over time.

The file-access model has an implicit requirement: the PDF must already be shared with (or owned by) the service account. The public download path uses an unauthenticated `fetch`, but the OCR path uses authenticated Drive API. If the file is publicly shared but not explicitly shared with the service account, the `files.copy` call will fail with a permissions error that surfaces as a generic "Gagal melakukan OCR via Google Drive" message.

1. **Service account file access assumption** — `files.copy` requires the service account to have at least Viewer access to the source file. If the PDF is only publicly shared (anyone-with-link), the copy will fail. Either share files with the service account or use `drive.files.create` with media upload instead. (confirmed)
2. **Opaque Drive API error message** — When `files.copy` fails due to permissions, the error message is a generic wrapper around the Drive API error. Add a check for 403/404 status and return a message explaining that the file must be shared with the service account email. (confirmed)
3. **Auth instantiation per request** — `google.auth.GoogleAuth` and credential parsing happen on every invocation of the OCR path. At low volume this is fine, but under load it creates unnecessary overhead. Consider caching the auth client at module scope. (likely)
4. **Orphaned temp docs on delete failure** — The `finally` block swallows delete errors silently. If the service account loses delete permission or hits a quota, temp docs will accumulate with no visibility. Consider logging the delete failure. (possible — depends on monitoring setup)

## Service account file access model

The existing download path fetches PDFs via `https://drive.google.com/uc?export=download&id=...` — an unauthenticated public URL that works for "anyone with link" sharing. The new OCR path calls `drive.files.copy` with the service account's credentials, which requires the service account to have at least Viewer permission on the source file. These are two different access models operating on the same file ID.

If a user shares a PDF with "anyone with the link" but doesn't explicitly share it with the service account's email, the download succeeds but the OCR fallback fails. The error that surfaces is `Gagal melakukan OCR via Google Drive: <Drive API error>`, which won't help the user understand they need to share the file with the service account. A targeted check for 403/404 from the copy call would improve the user experience significantly.

## Credential handling and auth lifecycle

```javascript
const auth = new google.auth.GoogleAuth({
  credentials,
  scopes: ["https://www.googleapis.com/auth/drive"],
});
const drive = google.drive({ version: "v3", auth });
```

This creates a fresh auth client for every OCR request. The `googleapis` library internally caches tokens within an auth instance, but the instance itself is reconstructed each time. If this endpoint ever gets called in batch (processing multiple certificates), the repeated JSON parsing and auth initialization will add latency. Moving the auth client to module scope (initialized lazily on first use) would eliminate this.

## Temporary document cleanup

The silent swallow of `deleteErr` in the `finally` block means orphaned documents are invisible. In a service account's Drive, these accumulate indefinitely with no signal. A `console.warn` would provide minimal observability without disrupting the response.

```javascript
finally {
  if (newDocId) {
    try {
      await drive.files.delete({ fileId: newDocId });
    } catch (deleteErr) {
      // Ignore delete errors - no visibility into accumulating orphans
    }
  }
}
```

## Response shape asymmetry

The OCR path adds `method: "ocr"` to successful responses. The text-extraction path does not include a `method` field. Any client that conditionally renders based on the extraction method should handle the absence of this field for text-based extractions. If the field is useful to the client, adding `method: "text"` to the primary path would make the contract symmetric.
