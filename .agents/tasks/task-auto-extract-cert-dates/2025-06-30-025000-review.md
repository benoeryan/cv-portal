# PDF certificate date extraction via Google Drive download

The "Fetch dari Link" button now downloads PDF certificates from Google Drive, extracts text with `pdf-parse`, finds date patterns (`YYYY/MM/DD` or `YYYY-MM-DD`), takes the last one, and converts it to Japanese format (`YYYYå¹´MMæœˆDDæ—¥`). This replaces the previous stub that only populated cert names without dates. A new API route at `/api/extract-cert-date` handles the server-side PDF processing.

**Watch for:** Google Drive files that aren't publicly shared will fail silently at the user level with a generic error (confirmed). The extraction only handles 2 of the 4 previously-supported certificate links, dropping `sertifikatSenmonkyuu` and `sertifikatSelesaiMagang` coverage (confirmed). The "last date in the document" heuristic works for the Apps Script reference but is fragile for PDFs with different layouts (likely).

## High-level view

The API route is a straightforward port of the Apps Script logic: extract a Google Drive file ID from the URL, download via the public `uc?export=download` endpoint, handle the virus-scan interstitial for large files, parse the PDF buffer, regex-match all dates, and return the last one in Japanese format. No authentication path exists for restricted files.

The frontend replaces the old click handler wholesale. The old handler processed four certificate types and populated a `sertifikat` array. The new handler processes only two (JFT and SSW) and writes to individual date fields (`tanggalJFT`, `tanggalSSW`, `tanggalSSWKaigo`). This is a coverage regression for candidates who have `sertifikatSenmonkyuu` or `sertifikatSelesaiMagang` links.

The SSW-to-Kaigo logic copies the same extracted date to both `tanggalSSW` and `tanggalSSWKaigo` when `bidangKerja === "KAIGO"`. This assumes the SSW certificate PDF contains the Kaigo exam date too, which may not be correct if they're separate exams with separate certificates.

## Coverage regression: only 2 of 4 certificate types handled

The master branch handler checked `sertifikatBahasaJepang`, `sertifikatSSW`, `sertifikatSenmonkyuu`, and `sertifikatSelesaiMagang`. The new implementation checks only the first two. The `CERT_TEMPLATES` constant still defines entries for `tanggalShuryoShomei` (æŠ€èƒ½å®Ÿç¿’ä¿®äº†è¨¼æ˜Žæ›¸) and the `sertifikatSelesaiMagang` link exists in the data model and form, but the extraction button ignores them. A candidate with a Senmonkyuu PDF link will get no date auto-filled for that row.

The Apps Script equivalent also processes a "column BG" for Kaigo separately from the general SSW link. The current implementation assumes they share the same URL, which diverges from the reference behavior.

## Google Drive access model

The download path uses the unauthenticated `https://drive.google.com/uc?export=download&id=...` endpoint. This works only for files with "Anyone with the link" sharing. The virus-scan interstitial handling (extracting a `confirm` token from the HTML) covers larger files, but the token extraction regex (`/confirm=([a-zA-Z0-9_-]+)/`) may not match all variants Google serves â€” their interstitial page format changes periodically.

The Apps Script uses `DriveApp.getFileById()` which runs with the script owner's OAuth credentials and can access any file the owner has permission for. The web portal version is strictly weaker: any file that isn't publicly shared will fail with a generic "file tidak dapat diakses" error.

## SSW-to-Kaigo date duplication

```javascript
if (data.bidangKerja === "KAIGO") {
  handleChange("tanggalSSWKaigo", result.date);
  results.push(`SSW Kaigo: ${result.date}`);
}
```

This copies the SSW exam date to the Kaigo field without a separate extraction. If the Kaigo exam has its own certificate (as suggested by the Apps Script having a separate column BG URL), this produces incorrect data in the CV.

## Information leakage in error response

When no dates are found, the API returns:

```javascript
{ success: false, error: "...", rawText: pdfText.substring(0, 200) }
```

This `rawText` field exposes the first 200 characters of the parsed PDF to the browser. Certificate PDFs contain candidate names, birth dates, and exam details. The same field is also returned on success. Remove `rawText` from production responses or gate it behind a debug flag.

## Date extraction heuristic

The "take the last date" strategy assumes exam dates appear after birth dates in Japanese testing certificates. A certificate with an expiry date or issuance date positioned after the exam date would extract the wrong value. No validation is done on whether the extracted date is plausible (not in the future, not before 2000).
